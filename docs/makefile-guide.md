# Makefile 组织与编排规范

## 目标

这份文档定义仓库内 Makefile 的长期约束，重点回答 4 个问题：

- 根 `Makefile` 应承担什么职责
- 什么类型的命令可以下沉到子目录
- 共享变量应如何管理
- 新增命令时如何判断归属

本文档只描述长期约束，不规定某次具体重构的拆分方案、迁移顺序或验收目标。

## 根入口原则

- 仓库必须保留统一的根 `Makefile` 入口，承接常用开发、测试、构建和运行时命令。
- 根 `Makefile` 负责跨模块编排、共享变量收口和稳定命令名暴露。

根 `Makefile` 适合承载的命令包括：

- 跨多个模块的聚合命令
- 依赖统一环境变量的系统级编排命令
- 面向仓库使用者的稳定入口命令

## 命令归属原则

新增命令时，按以下顺序判断归属：

1. 这个命令是否在编排整个系统？
2. 这个命令是否横跨多个模块？
3. 这个命令是否依赖统一的全局环境变量？
4. 这个命令是否天然只属于某一个目录？

归属规则：

- 系统级编排命令应优先保留在根 `Makefile`
- 跨模块聚合命令应优先保留在根级
- 强领域、局部化、在原目录执行更自然的命令，可以放到子目录 `Makefile`
- 纯工具原生命令若没有统一编排价值，不应为了包装而机械新增 Make 目标

## 子目录 Makefile 原则

子目录 `Makefile` 不是必需品，只在以下情况引入：

- 命令天然属于该目录
- 命令主要服务该目录内的开发工作流
- 该命令实现细节较重，继续堆在根 `Makefile` 会明显降低可维护性
- 根目录与该目录需要复用同一套实现，且复制成本已明显偏高

引入子目录 `Makefile` 时，必须满足：

- 根目录仍保留统一入口
- 根目录与子目录不得复制同一套实现
- 共享变量语义保持一致

## 共享变量原则

- 共享变量必须有单一权威来源。
- 同一变量在根目录与子目录中的业务含义必须一致。
- 环境派生逻辑不得在多个文件中重复维护。

重点变量包括但不限于：

- `PLATFORM`
- `ENV`
- `TEST_NAME`
- `JAVA_HOME`
- 端口类变量
- 路径类变量

- 不允许对同名变量赋予不同业务语义
- 新增局部变量时，应尽量使用领域前缀，避免与全局变量冲突
- 测试相关变量应保持统一前缀风格，例如 `MAZE_TEST_*`

## 命名原则

- 目标名优先使用动词或动宾结构
- 同类命令应保持对称命名
- 代理命令命名应直接表达代理对象
- 破坏性命令必须显式命名，不能伪装成普通操作

- `build-go`
- `check-frontend`
- `test-integration`
- `proxy-db`
- `destroy`

## 帮助信息原则

- 所有面向使用者的目标都必须带 `##` 注释
- `make help` 输出必须足够让使用者理解命令用途
- 若命令属于破坏性操作，说明中必须显式标注
- 若命令只适用于某个平台或环境，说明中也应明确体现

## Shell 编写原则

- 默认使用 `@` 抑制不必要噪音
- 关键步骤必须输出可读日志
- 多步流程必须失败即退出，不允许静默吞错
- 复杂逻辑必须有解释“为何这样做”的注释
- 不要在多个命令块中复制同一套环境派生、等待、清理逻辑

## 可维护性原则

- 根 `Makefile` 必须保持按职责分段，避免不同类型命令交叉散落
- 新增命令时，应优先复用已有变量与辅助逻辑
- 当某个区块已经明显失控时，可以进行结构性拆分，但拆分方案属于实现设计，不属于本规范

## 禁止事项

- 不要把破坏性清理伪装成普通 `down`
- 不要在根和子目录重复维护同一套实现
- 不要让根目录与子目录对同名变量含义不同
- 不要为了“看起来模块化”而无收益地增加文件层级
